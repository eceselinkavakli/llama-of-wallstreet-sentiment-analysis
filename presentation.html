Methodology

⚙️ Pipeline Architecture

  1. Data ingestion: load Reddit comments via Pandas, parse timestamps
  2. Combined LLM call: single structured-output request returns both ticker mentions and sentiment
  3. Parallel processing: 8 concurrent workers via ThreadPoolExecutor
  4. Filtering: drop comments with no ticker mentions; explode multi-ticker comments into one row per ticker
  5. Daily aggregation: mean / min / max / std sentiment per (date, ticker)
  6. Visualisation: Matplotlib dual-axis trend charts (sentiment + comment volume)
  7. SLURM agent: natural-language interface for submitting the job at full scale
🎯 Goal: let a data scientist run large-scale LLM sentiment analysis on the Leonardo supercomputer without writing any code themselves.
Methodology

🧩 Structured Output Design

We deliberately combined ticker extraction and sentiment classification into a single LLM call using a Pydantic schema, instead of two separate calls.

Why one call?

  • Halves API cost and latency vs. two sequential calls
  • Avoids inconsistency between extraction and sentiment steps
  • Pydantic + Enum gives type-safe, validated output — no regex parsing

Sentiment → score mapping

  • very_positive → +2
  • positive → +1
  • neutral → 0
  • negative → −1
  • very_negative → −2
Model: Mistral-Small-3.2-24B-Instruct, served locally via vLLM on the Leonardo cluster (OpenAI-compatible API, accessed through LangChain).
Methodology

⚡ Parallelism & Error Handling

Concurrency

  • ThreadPoolExecutor with 8 workers
  • Each worker issues one LLM request per comment
  • Results re-assembled in original row order via index tracking
  • Progress logged every 100 comments with live throughput/ETA

Resilience

  • Each comment gets one retry after a 2s backoff on failure
  • Second failure → safe fallback: empty tickers, neutral sentiment
  • Long comments truncated to 1,500 characters before sending
  • Pipeline never crashes on a single bad comment
⚠️ Trade-off: the fallback-to-neutral strategy is simple and robust, but silently masks failures — a comment that errors twice looks identical to one that was genuinely neutral. We revisit this in Critical Evaluation.

SLURM Agent

🤖 Agent Architecture

Lets non-technical users run and monitor the full-scale (100k-comment) job on Leonardo using plain English.

ToolPurposeUnderlying command
submit_job()Submit job to SLURMsbatch
check_status()Check job statesqueue / sacct
cancel_job()Cancel running jobscancel
fetch_results()Download output filesrsync / scp
list_recent_jobs()Show past jobssacct
Architecture pattern: ReAct loop — the same Mistral model that runs sentiment analysis reads user intent, picks a tool, executes it via LangChain function calling, and reports back in plain English.
SLURM Agent

💬 Agent in Action

User: "Hi! Can you run the sentiment analysis pipeline for me?"
Agent: calls submit_job("student_job_LLM.py") → "Done! Job ID: 1234567. Usually takes ~30 minutes for the full 100k comments."
User: "Is my job done yet?"
Agent: calls check_status(1234567) → "Still running — based on typical processing, results in about 10 more minutes."
User: "Get me the results"
Agent: calls check_status() then fetch_results() → "Downloaded to ./output/: results_daily_sentiment.csv and 6 PNG sentiment charts."
  • No code writing required by end users
  • Automatic confirmation before destructive actions (e.g. cancel_job)
  • Real implementation includes a local-mode simulation fallback when SLURM isn't available
Critical Evaluation

🔍 Limitations

  • Tiny relevant sample: only 48/1,000 comments (4.8%) mention a public company — most tickers have just 1–2 mentions, far too few for statistically reliable sentiment
  • No ground-truth validation: sentiment labels are never checked against human-annotated data, so we don't know the model's real accuracy
  • Silent failure fallback: a comment that fails twice is indistinguishable from a genuinely neutral comment in the output
  • Single-model dependence: all results rest on Mistral-Small's judgment with no second model for cross-checking or ensembling
  • Truncation at 1,500 characters may cut off sentiment-bearing content in longer comments
⚠️ Bottom line: the pipeline works end-to-end, but at this sample size the per-ticker numbers should be read as a proof of concept, not as trading-grade signals.
Critical Evaluation

🧭 Weak Points & Alternatives Considered

Weak points

  • No deduplication of repeated comments/bots
  • Ticker disambiguation is naive (e.g. "BA" could mean Boeing or British Airways context)
  • Daily aggregation has many single-comment days → noisy mean/std
  • No confidence score returned alongside sentiment

Alternatives we considered

  • Two separate calls (extraction, then sentiment) — rejected: 2× cost/latency
  • Lexicon-based sentiment (e.g. VADER/FinBERT) — rejected: weaker on sarcasm/slang common in Reddit finance threads
  • Larger model (70B+) — rejected: throughput too slow for 100k-comment scale on shared HPC time
  • Async I/O instead of threads — considered, but ThreadPoolExecutor was simpler given I/O-bound vLLM calls
Discussion

🛠️ Challenges & Improvements

Challenges we faced

  • Balancing parallelism (speed) against vLLM endpoint rate limits
  • Designing a prompt that reliably returns an empty ticker list for off-topic comments
  • Keeping the SLURM agent's tool calls safe (no destructive action without confirmation)

Improvements for next time

  • Run on the full 100k-comment dataset via SLURM batch, not just the 1,000-row sample
  • Add a held-out, hand-labelled validation set to measure real accuracy
  • Track and surface failed/retried comments instead of silently defaulting to neutral
  • Add confidence scores and ticker-disambiguation logic

Conclusion

🎓 Summary

What we built

  • 🤖 A combined ticker-extraction + sentiment pipeline using a local LLM with structured outputs
  • ⚡ Parallel processing (8 workers) with automatic retry and safe fallback
  • 📊 Daily and overall sentiment aggregation with trend visualisations
